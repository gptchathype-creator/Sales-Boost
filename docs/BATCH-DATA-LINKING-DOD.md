# Definition of Done: корректная связка batch -> звонок -> разбор

## Цель
Убрать приблизительную навигацию по имени автосалона и перейти к детерминированной связке:

`CallBatchJob -> CallBatchAttempt(voiceCallId) -> Audit/CallReview`

Чтобы кнопка "Открыть диалог" всегда открывала именно тот звонок, который относится к выбранному `job`.

## Что считается "готово" (DoD)

### 1) Backend-модель связи
- Для каждого `CallBatchJob` хранится ссылка на итоговый разбор звонка (например `auditId`) **или** гарантированно восстанавливаемая связь через `voiceCallId`.
- После завершения звонка webhook/обработчик записывает связь `job <-> audit` в БД.
- При ретраях сохраняется связь по последней успешной/финальной попытке согласно бизнес-правилу.

### 2) Backend API
- `GET /api/admin/call-batches/:id/jobs` возвращает для каждой строки:
  - `jobId`, `dealershipId`, `status`, `attempt`, `lastOutcome`
  - `linkedAuditId` (nullable)
  - `hasTranscript` (boolean)
- Для `linkedAuditId = null` API отдает понятный признак причины (например, "no_call_review_yet").

### 3) Frontend-навигация
- В `Проверки -> Автосалоны -> Detail batch` кнопка "Открыть диалог" открывает разбор строго по `linkedAuditId`.
- Если `linkedAuditId` отсутствует:
  - показывается русское состояние "Разбор еще не сформирован",
  - кнопка перехода неактивна,
  - нет fallback-переходов по названию салона.

### 4) UI/UX без дублей логики
- Маппинг статусов звонка и исходов вынесен в shared-утилиту (единый источник правды).
- Компоненты summary/статусов переиспользуются между `AuditDetail` и `AuditBatchDetail`.
- Нет параллельных "ручных" мапперов статусов в нескольких файлах.

### 5) Наблюдаемость и контроль качества
- Для каждого `job` логируется lifecycle: queued -> dialing/in_progress -> completed/failed/retry.
- Есть метрика "jobs with linkedAuditId / total completed jobs".
- Порог приемки: не ниже 99% для тестовых и реальных прогонов (исключая отмененные/прерванные случаи по правилам).

## Критерии приемки (Acceptance Criteria)

1. Для batch из 10+ звонков:
- у всех завершенных `job` есть корректный `linkedAuditId` или явная причина отсутствия.

2. Клик по "Открыть диалог" в любой строке batch:
- всегда открывает правильный audit/detail,
- не открывает "чужой" салон/чужой звонок.

3. В UI нет английских технических статусов:
- только русские бизнес-ярлыки.

4. При пустых реальных данных:
- preview mock остается только как visual fallback,
- но не подменяет data-linking в production режиме.

## Неготово (Out of Scope для этого этапа)
- Полный call-player с записью аудио в batch-detail.
- Автоматическая AI-оценка качества диалога вне текущего пайплайна.
- Изменение базовых backend-контрактов, не связанных с linking.

## Технический план внедрения

1. Backend:
- добавить `linkedAuditId` в DTO jobs API;
- записывать связь в момент финализации webhook.

2. Frontend:
- заменить текущий fallback-поиск по имени салона на строгое чтение `linkedAuditId`;
- унифицировать маппинг статусов в shared helper.

3. Тестирование:
- локальный test mode (mock calls) + реальный smoke с 3-5 номерами;
- чек-лист на корректность drill-down и статусов.
