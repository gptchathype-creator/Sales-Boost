# СПЕЦИФИКАЦИЯ МЕТРИК И СКОРИНГА

## Назначение

Документ фиксирует единую логику для двух аналитических таблиц:

- таблица автосалонов (реальные телефонные разговоры),
- таблица сотрудников (проверки в симуляторе).

Для каждого показателя описано:

- что он означает,
- как считается,
- откуда берутся данные,
- что уже реализовано,
- чего не хватает,
- как реализовать корректно.

Цель: убрать разночтения из-за моков и сделать цифры согласованными между UI, бэкендом и моделью данных.

---

## Политика источников данных

### 1) Источник метрик автосалонов

Таблица автосалонов должна строиться на реальной телефонии (`VoiceCallSession`), так как это операционное качество работы отдела продаж.

### 2) Источник метрик сотрудников

Таблица сотрудников должна строиться только на проверках симулятора (`TrainingSession`), так как реальные звонки сейчас нельзя надежно связать с конкретным сотрудником.

### 3) Единая шкала

Для обоих потоков используется одна шкала AI-оценки 0-100 и одна логика интерпретации качества:

- AI-рейтинг автосалона = агрегат по реальным звонкам,
- AI-рейтинг сотрудника = агрегат по симулятору.

---

## Роли и видимость

### Super Admin

Должен видеть иерархию во всех таблицах:

- Холдинг
- Автосалон
- Город

### Роль холдинга

Видит только свои салоны. Колонку «Холдинг» можно скрывать, если скоуп уже ограничен одним холдингом.

### Роль автосалона

Видит только свой салон. Колонка «Холдинг» не отображается.

---

## Целевая модель данных (техническая)

Чтобы реализовать метрики полностью корректно, нужны стабильные связи оргструктуры и телефонии.

### Основные сущности

- `Holding`
  - `id`
  - `name`

- `Dealership`
  - `id`
  - `holdingId` (FK -> `Holding`)
  - `name`
  - `city`

- `DealershipPhone`
  - `id`
  - `dealershipId` (FK -> `Dealership`)
  - `phoneE164` (unique)

- `User` (сотрудник)
  - `id`
  - `fullName`
  - `dealershipId` (FK -> `Dealership`) или отдельная таблица занятости

- `VoiceCallSession` (уже есть)
  - `callId`
  - `to`
  - `outcome`
  - `durationSec`
  - `transcriptJson`
  - `evaluationJson`
  - `totalScore`
  - `answerTimeSec` (целевая колонка, сейчас отсутствует)

- `TrainingSession` (уже есть)
  - `userId`
  - `status`
  - `completedAt`
  - `totalScore`
  - `evaluationJson`

### Обязательные связи

- `VoiceCallSession.to` -> `DealershipPhone.phoneE164` -> `Dealership.id`
- `User.dealershipId` -> `Dealership.id`
- `Dealership.holdingId` -> `Holding.id`

Без этих связей метрики салонов/сотрудников не будут полностью достоверными.

## База скоринга: из чего складывается оценка

Атомарная оценка (0-100) формируется на уровне одной оцененной сессии.

Текущий пайплайн использует взвешенный чеклист и детерминированные штрафы.

Текущие блоки чеклиста:

- INTRODUCTION
- SALON_NAME
- CAR_IDENTIFICATION
- NEEDS_DISCOVERY
- INITIATIVE
- PRODUCT_PRESENTATION
- CREDIT_EXPLANATION
- TRADEIN_OFFER
- OBJECTION_HANDLING
- NEXT_STEP_PROPOSAL
- DATE_FIXATION
- FOLLOW_UP_AGREEMENT
- COMMUNICATION_TONE

На итог влияют:

- статусы пунктов чеклиста (YES/PARTIAL/NO/NA),
- вес каждого пункта,
- штрафы (например misinformation, no next step, passive style, early fail).

Тот же принцип шкалы 0-100 используется и для звонков, и для симулятора. Разница только в источнике данных.

## Периоды и динамика

### Поддерживаемые периоды

- Неделя (7d)
- Месяц (30d)
- Всё время

### Единое правило динамики

`Динамика = Метрика_текущий_период - Метрика_предыдущий_эквивалентный_период`

Примеры:

- Неделя: последние 7 дней vs предыдущие 7 дней
- Месяц: последние 30 дней vs предыдущие 30 дней

Для режима «Всё время» нужно зафиксировать одно правило и использовать его везде:

- либо показывать `—` (рекомендуется, чтобы не вводить в заблуждение),
- либо сравнивать с предыдущим календарным периодом по утвержденной политике.

### AI-рейтинг (автосалон)

**Описание**  
Среднее качество реальных звонков салона за выбранный период.

**Расчет**  
`AI_салона = round(avg(totalScore))`, где:

- звонок сопоставлен салону,
- `totalScore IS NOT NULL`,
- дата сессии входит в выбранный период.

**Источник данных**  
`VoiceCallSession.totalScore`, время сессии, сопоставление телефона с салоном.

**Текущее состояние**  
В UI часто подмешиваются моки; не везде есть стабильный агрегат из реальных сессий по салону.

**Как реализовать**  
1. Ввести надежное сопоставление `phone -> dealershipId`.
2. Сделать backend-агрегацию по периоду.
3. Отдавать в endpoint метрик салонов.

---

### Дозвон, %

**Описание**  
Доля успешных разговоров среди всех попыток звонка.

**Расчет**  
`Дозвон_% = 100 * count(outcome='completed') / count(all)`

для салона и выбранного периода.

**Источник данных**  
`VoiceCallSession.outcome` + период + сопоставление номера к салону.

**Текущее состояние**  
Платформенный расчет есть, но с разбивкой по салону часто мок/частичный контур.

**Как реализовать**  
Считать на том же агрегирующем endpoint, что и AI-рейтинг салона.

---

### Время ответа

**Описание**  
Среднее время до первого ответа менеджера в успешных звонках.

**Расчет**  
`Время_ответа = avg(answerTimeSec)`

для звонков с `outcome='completed'` и непустым `answerTimeSec`.

**Источник данных**  
Целевое поле: `VoiceCallSession.answerTimeSec`.

**Текущее состояние**  
Надежно не реализовано; поля в текущей схеме нет.

**Как реализовать**  
1. Добавить nullable колонку `answerTimeSec`.
2. Заполнять из событий телефонии/таймингов транскрипта.
3. Включить в агрегат салона.

---

### Количество сотрудников

**Описание**  
Сколько сотрудников прикреплено к салону.

**Расчет**  
`count(active employees by dealership)`.

**Источник данных**  
`User.dealershipId` (или таблица занятости).

**Текущее состояние**  
В UI часто прокси-значения из моков.

**Как реализовать**  
Добавить полноценную связь сотрудник -> салон и считать на бэкенде.

---

### Динамика

**Описание**  
Изменение AI-рейтинга салона относительно предыдущего равного периода.

**Расчет**  
`Динамика = AI_текущий - AI_предыдущий`.

**Источник данных**  
Два агрегата AI из соседних временных окон.

**Текущее состояние**  
В ряде экранов упрощенный/моковый тренд.

**Как реализовать**  
Считать на backend (оба окна) и возвращать числовую дельту.

**Отображение**  
- `+x.x` — рост,
- `-x.x` — падение,
- `—` — недоступно.

---

### Топ-ошибка (автосалон)

**Описание**  
Самая частая проблема в оцененных звонках салона за период.

**Расчет**  
1. Разобрать `evaluationJson.issues[].issue_type` по всем звонкам салона в периоде.
2. Посчитать частоту каждого `issue_type`.
3. Взять топ-1 (для детальной страницы можно топ-N).

**Источник данных**  
`VoiceCallSession.evaluationJson`.

**Текущее состояние**  
Как стабильный реальный агрегат обычно не реализован.

**Как реализовать**  
Варианты:

- парсить JSON на лету,
- или денормализовать в таблицу `CallIssue(callSessionId, issueType, severity)` при записи оценки.

**Набор ошибок (из текущего evaluator):**

- `NO_INTRO`
- `NO_SALON_NAME`
- `NO_NEEDS_DISCOVERY`
- `WEAK_PRESENTATION`
- `NO_NEXT_STEP`
- `NO_DATE_FIX`
- `WEAK_TRADEIN`
- `WEAK_CREDIT`
- `BAD_TONE`
- `PASSIVE_STYLE`
- `MISINFORMATION`
- `REDIRECT_TO_WEBSITE`
- `LOW_ENGAGEMENT`
- `PROFANITY`

---

### Статус (автосалон)

**Описание**  
Сводный индикатор состояния салона.

**Рекомендуемая логика**

- `Нет данных`: звонков меньше минимального порога достоверности
- `Критично`: низкий AI и/или очень низкий дозвон
- `Риск`: средний AI или отрицательная динамика
- `Норма`: стабильные хорошие метрики

**Базовые пороги (можно калибровать):**

- `Норма`: `AI >= 80` и `Дозвон >= 70`
- `Риск`: `AI 60..79` или `Дозвон 50..69` или отрицательная динамика
- `Критично`: `AI < 60` или `Дозвон < 50`
- `Нет данных`: `CallCount < MinCallsForStatus`

**Источник данных**  
Вычисленные агрегаты салона.

**Текущее состояние**  
Пороговая логика есть в mock-функциях, но часто на нереальных входных данных.

**Как реализовать**  
Считать статус на бэкенде от реальных агрегатов + минимального объема данных.

Средний результат сотрудника по симулятору за выбранный период.

`AI_сотрудника = round(avg(TrainingSession.totalScore))`, где:

### Динамика (сотрудник)

Изменение AI сотрудника относительно предыдущего равного периода.

`Динамика = AI_текущий - AI_предыдущий`.

### Количество проверок

**Описание**  
Сколько симуляторных проверок сотрудник завершил за период.

**Расчет**  
`count(TrainingSession)` по критериям периода/завершенности.

**Источник данных**  
`TrainingSession`.

**Текущее состояние**  
Сейчас часто мок.

**Как реализовать**  
Отдавать сырое count в employee-агрегатах.

---

### Провалы

**Описание**  
Сколько проверок сотрудник не прошел по качественному порогу.

**Базовое правило (рекомендуемое):**

- провал, если `totalScore < FailThreshold`.

**Альтернатива:**

- провал, если `status='failed'` ИЛИ `totalScore < threshold`.

**Источник данных**  
`TrainingSession.totalScore`, `TrainingSession.status`.

**Текущее состояние**  
Сейчас как правило моковая эвристика.

**Как реализовать**  
Зафиксировать одно продуктовое правило и применять серверно.

---

### Коммуникация

**Описание**  
Метка качества коммуникации сотрудника по оценке симулятора.

**Как считать**

Маппить сигналы из `evaluationJson` в категории, например:

- `PROFANITY` / `BAD_TONE` -> высокий риск коммуникации,
- `LOW_ENGAGEMENT` -> слабая вовлеченность,
- иначе -> норма.

**Источник данных**  
`TrainingSession.evaluationJson` (`issues`, `checklist`, доп. сигналы).

**Текущее состояние**  
В текущей таблице обычно моковый `communicationFlag`.

**Как реализовать**  
1. Зафиксировать финальный набор категорий.
2. Зафиксировать приоритеты конфликтов.
3. Считать на backend и (опционально) хранить денормализованную метку.

---

### Топ-ошибка (сотрудник)

**Описание**  
Самая частая ошибка сотрудника за период в симуляторе.

**Расчет**  
Mode по `issue_type` из всех `evaluationJson` сотрудника за период.

**Источник данных**  
`TrainingSession.evaluationJson.issues[].issue_type`.

**Текущее состояние**  
В таблице часто случайное/моковое значение.

**Как реализовать**  
Тот же подход, что для салона: парсинг на лету или денормализация issue-строк.

---

### Статус (сотрудник)

**Описание**  
Сводный статус сотрудника по симулятору.

**Базовая логика:**

- `Нет данных`: нет проверок в периоде
- `Критично`: низкий AI и/или много провалов и/или тяжелые сигналы коммуникации
- `Риск`: средний AI или часть провалов
- `Норма`: высокий AI без рисков

**Базовые пороги (можно калибровать):**

- `Норма`: `AI >= 80` и `Провалы = 0`
- `Риск`: `AI 60..79` или `Провалы 1..2`
- `Критично`: `AI < 60` или `Провалы >= 3`
- `Нет данных`: `CheckCount = 0`

**Источник данных**  
Вычисленные агрегаты сотрудника.

**Текущее состояние**  
Есть mock-логика, не привязана жестко к реальным агрегатам симулятора.

**Как реализовать**  
Считать статус на backend через общий конфиг порогов.

## Честная картина текущего состояния

### Что уже реально работает

- Есть хранение `VoiceCallSession` и evaluator-пайплайн.
- Для оцененных звонков сохраняются `totalScore` и `evaluationJson`.
- В `TrainingSession` есть поля оценки (`totalScore`, `evaluationJson`).

### Что смешано с моками

- В super-admin таблицах часто используются fallback-данные из mock layer.
- Агрегация звонков по салонам не везде считается строго из real sessions.
- Таблица сотрудников super-admin пока во многом mock-driven.

### Что критично доделать

1. Канонические сущности `Holding`/`Dealership`.
2. Таблицу сопоставления `phone -> dealership`.
3. Связь `User -> Dealership`.
4. Поле `answerTimeSec` в звонках.
5. Отдельные backend endpoints агрегатов для обеих таблиц.
6. Единый конфиг порогов статусов/провалов.
7. Реальную агрегацию top-ошибок из payload оценок.

## Рекомендуемый план реализации

### Этап 1. Усиление модели данных

- Добавить `Holding`, `Dealership`, `DealershipPhone`.
- Добавить связь сотрудника с салоном (`User.dealershipId` или отдельная занятость).
- Добавить `VoiceCallSession.answerTimeSec` (nullable).

### Этап 2. Backend-агрегаторы

Сделать endpoints:

- `GET /api/admin/analytics/dealerships?period=7d|30d|all`
- `GET /api/admin/analytics/employees?period=7d|30d|all`

Оба endpoint должны отдавать:

- метрики,
- динамику,
- статус,
- топ-ошибку,
- сырые count-поля, из которых получены агрегаты.

### Этап 3. Переключение фронта с моков

- Перевести таблицы на новые endpoints.
- Оставить mock fallback только в явном dev-режиме.
- Включить ролевую видимость колонок (холдинг показывать только где нужно).

### Этап 4. Валидация и контроль качества

- Тесты формул (unit/integration).
- Сверка агрегатов с выборкой сырых сессий.
- Продуктовое утверждение порогов и fail-правил.

## Краткий справочник формул

### Автосалоны

- `AI-рейтинг = avg(call.totalScore)`
- `Дозвон% = 100 * completed_calls / all_calls`
- `Время ответа = avg(call.answerTimeSec for completed_calls)`
- `Динамика = AI_current - AI_previous_window`
- `Топ-ошибка = mode(issue_type in call evaluations)`

### Сотрудники

- `AI-рейтинг = avg(training.totalScore)`
- `Количество проверок = count(training checks)`
- `Провалы = count(training checks by fail-rule)`
- `Динамика = AI_current - AI_previous_window`
- `Топ-ошибка = mode(issue_type in simulator evaluations)`

## Финальное правило согласованности

- Таблица автосалонов = качество реальной телефонии.
- Таблица сотрудников = качество симулятора.
- Шкала оценки единая (0-100), но потоки данных разные.
- Метрики сопоставимы на уровне управления, но не взаимозаменяемы по источнику.

Это правило должно быть явно отражено в продуктовых текстах и аналитической документации.
